## Answering your question: would multiple agents speed this up?

**For this task — no, it'd be negligible-to-negative.** The honest reasoning:

1. **The bottleneck is one shared, metered resource.** Everything funnels through the single Bloomberg Desktop API connection (`bbcomm`) on this one terminal, against one entitlement/data cap. Parallel agents would each open a blpapi session **contending for the same pipe and the same quota** — total throughput doesn't rise, but the cap burns *faster and concurrently*, which defeats the frugality mandate and my ability to watch consumption and stop before the lab limit.

2. **The pipeline is sequential and stateful:** pull delta → merge onto the large existing CSV → dedupe → commit → push. The per-chunk `bdh` already batches 30 tickers/request; the Bloomberg server serializes those regardless of how many agents ask.

3. **Wall-time is dominated by Bloomberg server response, not my orchestration.** Local merges of ~1M-row CSVs are fast (seconds). There's little parallelizable CPU work to win back.

4. **Parallel writes to one working tree risk corruption;** isolating git worktrees adds setup cost for zero gain when the data API — not the disk — is the constraint.

Where a fan-out *does* pay off (and why I'd reach for it elsewhere): many **independent** network/CPU jobs — e.g., scraping many unrelated web sources, or running independent test suites. Here, every step depends on the same single metered API, so I deliberately ran serially. That same logic is why I didn't launch a Workflow for the pulls.
